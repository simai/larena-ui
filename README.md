# Larena UI

Developer vertical slice: canonical `ui.button`, `ui.input` and `ui.dataview`
manifests are registered by `SmartRegistry`; real admin collections render the
canonical `ui.dataview` path with assets activated through `core.assets`. This
is not a full Simai Framework runtime or production-readiness claim.

Universal UI/frontend package for Larena smart components, backend-first rendering, hydration, design packs, UI asset graph, component registry, playground and frontend behavior contracts.

## Smart Registry and reference contract

`SmartRegistry` is the canonical discovery point for generic UI manifests and
package contributions. The first reference-ready component is `ui.button`,
rendered by `SmartManager` as the pinned Simai Framework `<sf-button>` element.
`SmartCatalogProjection` turns visible, complete manifests into localized typed
entries, and `SmartAiCatalogProjection` serializes the same entries as stable
machine-readable JSON. These projections never register components, invent
props or become a second source of truth.

```php
use Larena\Ui\Facades\Smart;

$artifact = Smart::render(
    'ui.button',
    [
        'text' => 'Save',
        'size' => '1',
        'type' => 'default',
        'scheme' => 'primary',
        'loading' => false,
        'disabled' => false,
        'native-type' => 'button',
        'aria-label' => 'Save',
    ],
    $assetActivation,
);
```

The facade is the concise application call surface and resolves the
container-owned `SmartManager`; the manager remains the DI-first implementation
and testing boundary. The legacy `Larena\Ui\Smart::render('sf-*')` adapter stays
available for existing low-level frontend-tag callers.

Reference controls, presets and examples are derived from the same manifest.
`SmartComponentReference` normalizes allowlisted controls, while
`SmartInvocationExampleBuilder` produces inert PHP and frontend examples. It
does not evaluate generated code or accept arbitrary classes and templates.
Reference controls auto-submit the reproducible GET form; preview and code are
still generated by the backend rather than synthesized in JavaScript.

Every catalog entry reports render, data-binding and effect readiness
independently. A structured prop that cannot be represented truthfully as a
custom-element attribute produces an explicit unavailable frontend example;
it is never flattened or guessed. Provenance exposes only package-relative
`resources/smart/.../manifest.json` paths and content hashes.

The current implementation is developer-testable and does not claim
production readiness or readiness of all Larena packages.

Canonical specifications are in `simai/larena-specs`.
